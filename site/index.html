Why DevinimJS

Keep deployment boring. Keep the UI reactive.

Upload files to any host, serve them from a CDN, and let native ES modules do the rest.

1

No consumer build

Use native <script type="module">. Your users never install Node, npm, Vite or Webpack.

2

Works with your CSS

Light DOM means CKCSS and your existing global stylesheet apply naturally to every component.

3

Small by default

The core runtime stays below 4 KB min+gzip. Async data, forms and routing are optional modules.

4

Server-friendly

HTML or PHP can provide initial state with data-*; components emit simple bubbling dv:* events.

5

Safe rendering defaults

Template interpolation escapes values by default, keeping the raw HTML boundary explicit.

6

Predictable structure

One component, one file, one clear API. Contracts, templates and a generator keep implementation consistent.

One pinned import

Start from a static HTML file.

Use exact versions in production. The first import provides component authoring primitives; individual component modules register themselves.

Production rule: pin a tagged version, never main or an unversioned URL.
<link rel="stylesheet" href="https://cdn.jsdelivr.net/gh/kayacuneyd/ckcss@v0.1.0-beta.1/dist/ckcss.min.css">

<dv-counter data-start="5" data-step="2"></dv-counter>

<script type="module" src="https://cdn.jsdelivr.net/gh/kayacuneyd/devinimjs@v0.2.0/dist/modules/dv-counter.js"></script>
