Learn by building

One product, ten deliberate steps.

Build a small task-and-notes workspace as the concepts accumulate. Every step stays usable in a browser; backend work is marked when persistence or shared data begins.

See the pathRead the API contracts

Project path

  1. 01

    Counter: the smallest component

    Write a .dv.js file with component(), typed props, reactive state, an action and an escaped html`` view.

    Outcome: a working counter custom element.

    Use the authoring contract
  2. 02

    Task list: state and events

    Model task creation, completion and removal in component state. Emit a bubbling dv:change event when the component’s public result changes.

    Outcome: an interactive task list with no server.

  3. 03

    Notes: persistence at the page boundary

    Keep the component focused on UI behavior. Let page code connect its emitted data to browser localStorage; this is a client-only prototype, not shared persistence.

    Outcome: notes survive a refresh in one browser.

  4. 04

    Form: values, errors and submit state

    Use createForm() for values, field errors, dirty tracking and the async submission lifecycle. Keep the submit handler in application code.

    Outcome: a validated profile or note form with clear feedback.

    Read the form guide
  5. 05

    Shared store: one cart or workspace state

    Introduce createStore() when multiple components need the same state. Subscribe from components and release external resources with the supported lifecycle helpers.

    Outcome: task summary, filters or cart count stay synchronized.

  6. 06

    API screen: async loading and errors

    Replace local fixtures with createAsyncState() and fetchJson(). Render loading, success, empty and error states explicitly.

    Boundary: backend-dependent. The API, response shape and authorization belong to the project.

    Read the async guide
  7. 07

    CRUD workspace: table, filter and pagination

    Compose the existing data-table, field, modal, confirm and toast components around the API flow. Keep server mutations in the page/application layer.

    Boundary: backend-dependent for real records; client-only with fixtures for the UI exercise.

    Browse component contracts
  8. 08

    Product flow: catalog to cart

    Use product-card and cart components with shared state. Add search or autocomplete only after the static product flow is clear.

    Boundary: the cart interaction can be client-only; inventory, prices, accounts and checkout require a trusted backend.

  9. 09

    Hash-routed application shell

    Use createHashRouter() for static-hosting-friendly routes such as #/tasks and #/settings. Preserve ordinary links and accessible headings.

    Outcome: a multi-view client application without server rewrite rules.

    Read the routing guide
  10. 10

    Ship the same assets

    Deploy the browser modules and CKCSS assets to static hosting, shared hosting, PHP or a VPS. For PHP, print escaped data-* attributes and load the component module.

    Boundary: deployment does not create an API; add the backend separately when the product needs persistence, identity or uploads.

    Read the PHP integration guide

What comes after the path?

A professional WYSIWYG editor is not a current DevinimJS component. Treat it as planned product work: define the document model, selection behavior, paste sanitization, undo/redo and upload contract before teaching it as a supported tutorial.

For the current release, practice the editor boundary with a safe preview of sanitized content rather than claiming a finished editor API.
