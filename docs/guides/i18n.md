# i18n / locale primitive

The contract in one sentence: **`t(this, key, fallback)` resolves through `data-*` override →
active-locale bundle entry → hardcoded fallback**, and each component owns its own bundle file.
See `adr/0019-i18n-locale-primitive.md` for the full design rationale; this guide is the
copy-paste path to wire one more component.

`dv-modal`, `dv-confirm`, `dv-cart` are the reference implementations — read one of them
alongside this guide if anything is unclear.

## 1. Is the string actually translatable copy?

Not every `this.str(key, fallback)` call is UI text. Some are behavioral configuration that
happens to be read the same way — e.g. `dv-field`'s `control` (`'input'`/`'textarea'`/`'select'`)
selects which HTML element to render, and `type` is a literal `input[type]` value; `dv-state`'s
`state` (`'empty'`/`'loading'`/`'error'`) selects a render branch. **Don't wire these** — a
"translation" for an HTML input type is a bug, not a feature. Only wire strings a human actually
reads on the page (labels, messages, aria-labels).

## 2. Add the component's locale bundle (a new sibling file)

Create `src/components/dv-your-component.locale.js`:

```js
/**
 * @module components/dv-your-component.locale
 * `dv-your-component`'s own locale bundle (ADR-0019) — co-located, not centralized, so wiring
 * another component never touches this file. `en` matches the component's pre-existing
 * hardcoded defaults verbatim.
 */
export default {
  en: {
    label: 'Existing English default, unchanged',
  },
  tr: {
    label: 'Türkçe karşılığı',
  },
};
```

Keep `en`'s values **byte-for-byte identical** to today's hardcoded fallback strings — this task
type is a translation-system wiring, not a wording change.

## 3. Register the bundle and subscribe to locale changes

In the component's own file:

```js
import { BaseComponent, html, define } from '../core/core.js';
import { t, registerLocales, onLocaleChange } from '../core/i18n.js';
import locales from './dv-your-component.locale.js';

registerLocales('dv-your-component', locales);

export class DvYourComponent extends BaseComponent {
  connected() {
    this.onCleanup(onLocaleChange(() => this.requestUpdate()));
  }
  // ...
}
```

`onLocaleChange()` fires synchronously whenever `setLocale()` is called anywhere on the page —
`requestUpdate()` is the existing "external data source" re-render escape hatch (already
documented on `BaseComponent`), reused here rather than inventing a second mechanism. If your
component already defines `connected()`, just add the one line to the existing method.

## 4. Replace `this.str(key, fallback)` with `t(this, key, fallback)`

```diff
- ${this.str('label', 'Save')}
+ ${t(this, 'label', 'Save')}
```

The `key` must be **camelCase** and identical to the bundle's key — this is the same
`dataset[key]` lookup `BaseComponent#str()` already uses (ADR-0005): `data-confirm-label` reads
as `confirmLabel`, not the literal string `'confirm-label'`. (Two pre-existing call sites got this
wrong before this primitive existed — see ADR-0019's "bug fix, in scope" note — check any `str()`
call you're porting for the same mistake while you're there.)

## 5. Parameterized strings

If the string interpolates a value (e.g. a per-row name), use `{placeholder}` tokens and pass a
`params` object:

```diff
- aria-label="Decrease ${item.name}"
+ aria-label="${t(this, 'decreaseLabel', 'Decrease {name}', { name: item.name })}"
```

```js
// bundle
tr: { decreaseLabel: '{name} azalt' } // translator controls word order, not just the value
```

Unmatched placeholders are left as literal text (not blanked), so a typo'd translation degrades
visibly instead of silently.

## 6. Tests to add

Mirror `tests/unit/dv-modal.test.js`'s i18n section (search that file for `ADR-0019`):

1. **Locale entry used, no override:** `setLocale('tr')`, `el.requestUpdate()`, assert the `tr`
   bundle text renders. Always `setLocale(null)` in a `finally` to reset the module-level override
   for later tests in the same file.
2. **`data-*` override still wins (regression):** set the `data-*` attribute *and* `setLocale('tr')`
   — assert the override text renders, not the bundle entry.
3. **Fallback still applies:** no override, no `setLocale()` call, default `document.lang` (`'en'`
   in the happy-dom fixture) — assert the hardcoded fallback renders (this should already pass
   unchanged from before wiring, since `t()`'s third tier is the exact old fallback).
4. **Parameterized substitution, two rows if applicable:** prove per-item values don't cross
   contaminate (see `tests/unit/atomic-components.test.js`'s `dv-cart` locale test for the
   pattern).

Also extend `examples/i18n.html` if your component makes sense in that live-switch demo (it isn't
required — the demo intentionally stays to the three original reference components, but nothing
stops a future task from adding a fourth section for its own component), and add or extend an
assertion in `tests/e2e/i18n.spec.js` if you do.

## 7. Don't do this

- Don't create or edit a shared `locale/en.js`-style file — bundles are per-component, always.
- Don't reach for `{plural, other {...}}` ICU syntax or a pluralization library — this primitive
  is deliberately simple substitution only (YAGNI). If a component genuinely needs plural forms,
  that's a new ADR, not an extension bolted onto this one.
- Don't add a `t()`-equivalent method to `BaseComponent` — it was measured and rejected in
  ADR-0019 specifically to keep `core.js` under its size gate. Import `t` from `../core/i18n.js`
  directly, exactly like the three reference components do.
